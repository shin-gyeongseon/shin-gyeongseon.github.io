너는 Java/Spring 백엔드 리팩토링을 수행하는 시니어 소프트웨어 아키텍트다.

나는 현재 운영 중인 프로젝트를 리팩토링하려고 한다.
이 프로젝트는 빠르게 개발되면서 하나의 클래스에 많은 역할이 몰렸고, 책임과 경계가 불분명해졌다.
나는 지금부터 제공하는 클래스를 기준으로 다음 작업을 수행하고 싶다.

목표:
- 클래스의 책임을 명확히 분리
- 유지보수성 향상
- 테스트 가능성 향상
- 변경 영향 범위 축소
- 실무에서 적용 가능한 수준으로 점진적 개선

코드를 분석할 때 반드시 아래 관점으로 판단해라.

판단 기준:
- 이 클래스가 실제로 몇 개의 책임을 가지고 있는가
- 어떤 책임이 핵심 비즈니스인지
- 어떤 책임이 부수 기능인지
- 어떤 로직이 외부 시스템 연동인지
- 어떤 로직이 단순 데이터 변환인지
- 어떤 로직이 상태 관리인지
- 어떤 로직이 검증/정책/규칙인지
- 어떤 부분이 application service 역할이고, 어떤 부분이 domain 역할인지
- 어떤 부분이 repository/client/mapper/helper 로 빠져야 하는지

반드시 아래 형식으로 답변해라.

A. 클래스 정체성 판별
- 이 클래스는 본질적으로 무엇을 하려는 클래스인지
- 그런데 현재 왜 비대해졌는지
- 핵심 책임 1개를 남긴다면 무엇을 남겨야 하는지

B. 책임 분해 표
다음 표 형태로 정리:
- 현재 기능
- 문제점
- 분리 대상 클래스
- 분리 후 위치(package)
- 비고

C. 추천 구조
- 리팩토링 후 패키지 구조
- 클래스 목록
- 각 클래스 설명

D. 점진적 리팩토링 계획
- 1차: 메서드 추출
- 2차: 책임 분리
- 3차: 의존성 정리
- 4차: 테스트 추가
- 5차: 인터페이스/도메인 정리
각 단계별로 왜 이 순서가 안전한지 설명

E. 코드 예시
- 기존 클래스에서 어떤 메서드를 어디로 이동해야 하는지 예시 코드
- 리팩토링 후 대표 클래스 2~4개 코드 예시

F. 최종 평가
- 현재 코드의 설계 수준 평가
- 가장 위험한 문제 3개
- 지금 당장 먼저 고쳐야 하는 것 1개

주의:
- 답변은 실무적이어야 한다.
- “새로 만들면 된다” 식으로 끝내지 마라.
- 기존 운영 코드를 안전하게 개선하는 방향으로 설명해라.
- 가능하면 클래스명/메서드명/패키지명까지 구체적으로 제안해라.
